#### 1. 정보보안 정책을 수립할 때 고려해야 할 주요 요소는 무엇인가요?
고려해야할 주요 요소들은 다음과 같습니다.

    보안 정책에 대한 목적과 범위 - 왜 정책이 필요한가? 정책이 누구에게 적용되는가?

    보안 정책에 관한 역할과 책임 - 개인 혹은 부서의 책임화. e.g. 관리자, 직원
    
    보안 정책에 관한 위험 평가 및 관리 - 잠재적 위협 확인 및 위협에 대한 대응 절차 정의

    보안 정책에 대한 데이터 분류와 처리 - 데이터 민감도에 따른 분류. e.g. confidential, public

    보안 정책에 대한 접근 제어와 인증 - 시스템, 네트워크, 데이터에 대한 접근과 인증 방식 정의. e.g. 패스워드 정책, 다단계 인증, 접근 권한

    보안 정책에 대한 인식과 교육 - 보안 모범 사례와 잠재적인 위협에 대한 정보를 지속적으로 제공 할수 있는 지속적인 보안 교육의 중요성.

    보안 사고관련 대응 및 보고 - 보안 사고에 대응할수 있는 프로토콜과 복구 및 해결 단 설정.

    보안 정책 관련 규정 준수와 법적 요구 사항 - 정책이 법률, 규정 및 산업 규정(e.g. GDPR)과 일치하는지에 대한 확인 

    보안 정책과 관련해 물리적 보안 설정 - 서버, 데이터 센터와 같은 물리적인 자산에 대한 무단 접근 또는 도난 방지.

    보안 모니터링과 감사 - (시스템, 네트워크, 데이터) 잠재적인 보안 침해 대비와 규정 준수 보장 및 취약점 확인

    보안 정책에 대한 문서화와 검토 - 지속적으로 변화하는 보안 위협및 기술에 적응하기 위해 보안 정책을 문서화, 유지관리, 검토 및 업데이트

    보안 정책에 대한 커뮤니케이션 - 모든 임직원 및 이해관계자에게 명확하게 보안 정책에 대한 전달과 정책 위반의 결과 설명.


#### 2. 시스템 보안의 기본 원리는 무엇인가요?
시스템 보안의 기본 원리는 다음과 같습니다.
    
    기밀성(Confidentiality) - 민감한 정보는 권한이 있는 사용자만 접근이 가능하도록 하여 데이터의 무단 접근/공개를 방지.
    
    무결성(Integrity) - 데이터가 미승인 사용자에 의한 무단 수정 방지 및 데이터의 정확성과 일관성 유지.
    
    가용성(Availability) - 권한이 있는 사용자만이 시스템, 서비스, 데이터 사용이 가능하고 접근이 가능. 리소스에 대한 지속적인 접근 보장. 
    
    인증(Authentciation) - 시스템에 접근하려는 유저 혹은 엔티티의 신원을 확인. e.g. 패스워드, 생체 인식, 다단계 인증
    
    인가(Authorization) - 사용자의 인증 이후, 사용자가 접근하거나 수행할수 있는 작업이나 리소스를 결정. e.g. 최소 권한 원칙에 따른 접근 제어
    
    로그 및 감사 추적(Accountability and Auditing) - 시스템 내 보안 위반 식별 사용자 활동 추적, 기록 및 기록 유지 
    
    탄력성과 중복성(Resilience and Redundancy) - 시스템 공격이나 장애의 상황에 직면하더라도 지속적인 작동을 보장하기 위해 탄력성과 중복성을 갖도록 설계가 되어야하는 원칙. e.g. 백업, 장애 복구
    
    최소 권한(Least Privilege) - 유저는 작업을 하기 위한 최소한의 필요한 권한과 허가를 받아야하는 원칙. 잠재적인 내부자의 위협과 보안 침해 가능성을 낮춤.
    
    단계별 보안/계층화된 보안(Defenece in Depth) - 시스템과 네트워크를 걸쳐 여러 계층의 보안 제어 구현을 해야하는 원칙.
    해당 원칙은 단일 실패지점이 발생할 가능성을 최소화.
    e.g. 방화벽 + 바이러스 백신 + 침입 탐지 시스템 등을 마련.
    
    보안 설계(Security by Design) - 설계 단계에서 시스템의 보안 고려 사항을 통합해야하는 원칙. 설계단계가 개발의 모든 단계(설계, 구현, 유지, 관리)에 영향을 주기때문에 모든 보안적인 영향을 고려해야한다.


#### 3. 시스템 취약점을 어떻게 식별하나요?
    [기술적인 부분]
    취약점 스캔 - Nessus, OpenVAS 또는 Qualys와 같은 자동화된 도구를 활용해서 시스템, 네트워크 및 애플리케이션에 알려진 취약점을 스캔.
    스캐너는 DB에 나열된 알려진 취약점과 시스템 구성 및 설치된 소프트웨어 버전을 비교.
    
    침투 테스트 - 시스템에 대해 통제되고 승인된 시뮬레이션 공격을 수행하여 취약점을 찾아냅니다. 해당 테스트는 도구와 기술을 사용하여 시스템의 약점을 이용하여 공격자의 잠재적 진입점을 식별.
    
    시스템 패치 관리 - 운영 체제, 소프트웨어 및 어플리케이션에 대한 보안 패치 및 업데이트를 정기적으로 적용. 패치를 통해 시스템을 최신 상태로 유지 및 잠재적 위험 완화
    
    네트워크 트래픽 분석 - Wireshark 또는 Splunk와 같은 도구를 사용하여 네트워크 트래픽을 모니터링과 분석을 하여 비정상적인 활동 탐지.
    
    어플리케이션/시스템 코드 검토 - 자동화된 도구가 놓칠수 있는 취약점을 시스템 코드를 검토를 통해 발견이 가능하다. e.g. 보안 결함에 대한 코드의 심층 분석
    
    보안 감사 및 평가 - 시스템의 구성, 접근 제어 및 로그 검토 및 평가를 정기적으로 수행함으로써 악용될수 있는부분이나 약점들을 식별이 가능하다.
    
    보안 정보와 이벤트 관리(SIEM) - 보안 감사 및 평가와 비슷하게 다양한 소스에서 로그 데이터를 수집 및 분석하는 도구를 구현.
    잠재적 취약점이나 비정상적인 패턴 혹은 이상 현상을 식별이 가능하다.

    [비기술적인 부분]
    자산/인벤토리 관리 및 위험 평가 - 각 자산에 대해 최신화 유지 및 자산에 대한 잠재적 취약점을 평가하여 자산에 대한 중요성을 우선순위화.

    소셜 엔지니어링 테스트 - 보안과 관련된 사람들의(직원, 관리자 등...) 인적 사항 평가.
    
    버그 포상금 프로그램 및 커뮤니티 리포팅 - 보안과 관련된 외부 전문가가 시스템에 대한 취약점을 식별하고 보고할수있도록 장려하는 방법


#### 4. 서버 취약점 점검을 한다고 했을 때, 가장 먼저 무엇을 어떻게 할 건가요?
서버 취약점 점검시에 우선적으로 현재 서버에 관한 모든 데이터와 상태를 스캔하여 정보들을 모아야합니다.
    OS, 설치되어있는 소프트웨어와 애플리케이션들, 네트워크 환경설정, 열려있는 포트와 서비스, 유저 계정과 권한, 보안 정책과 보안 설정, 모니터링과 로깅의 환경설정, 하드웨어 디테일, 이전 보안 사고나 패치 히스토리, 백업과 복구 환경설정 등등...


#### 5. 바이러스와 악성코드의 차이점은 무엇인가요?
바이러스는 합법적인 프로그램이나 파일 내에 자신을 첨부해 확산 및 복제하는 일종의 악성코드입니다. 일반적으로는 실행 파일, 스크립트 또는 문서에 같이 첨부되며, 감염된 이메일 첨부파일 열기 혹은 감염된 프로그램 실행과 같은 사람의 작업이 필요로 합니다. 바이러스가 작동하게 되면 파일 손상 또는 수정, 데이터 손실, 시스템 불안정과 같은 컴퓨터의 기능 및 보안의 손상이 발생할수 있습니다.

악성코드는 시스템이나 데이터를 손상,방해 혹은 무단으로 접근할수 있도록 설계된 다양한 유형의 유해 소프트웨어를 지칭합니다. 바이러스뿐만 아니라 웜, 트로이 목마, 랜섬웨어, 스파이웨어, 애드뒈어 등 같은 기타 악성 소프트웨어도 포함이 되며 다양한 목적을 가질수가 있습니다. e.g. 피싱 이메일, 감염된 웹사이트, 손상된 소프트웨어 다운로드, 시스템 취약점 악용.


#### 6. DoS(Denial of Service)와 DDoS(Distributed Denial of Service)의 차이점은 무엇인가요?
DoS(Denial of Service)는 하나의 단일 공격자가 서버에 수많은 트래픽을 범람시켜서 웹사이트, 서버 혹은 리소스를 사용자가 사용할수 없게 만드는 공격입니다.

DDoS(Distributed Denial of Service) DoS 공격과 유사하지만 다양한 위치에 있는 여러 대의 컴퓨터 또는 기계들을 동시에 사용하여 표적 리소스를 범람시키는 DoS 공격입니다.


#### 7. 랜섬웨어에 감염됐을 때 대처법은 무엇인가요?
    1. 랜섬웨어 감염된 증상과 랜섬웨어 방식 확인
        - 평소 문제없이 열렸던 문서, 사진, 그림 음악, 동영상, 파일들 중 일부 혹은 전체가 읽을 수 없게 되거나 아예 열리지 않는 현상이 있는가?
        - 파일 확장자혹은 파일 이름이 변경이 되어 있는가?
        - 평소 사용하던 운영체제를 부팅에 문제가 있었는가? 혹은 바탕화면에 변경사항이나 새로 만들어진 파일이 있는가?
        - 랜섬웨어의 종류가 알려진 방식인가? 현재 있는 복구툴로 복호화가 가능한가?

    2. 랜섬웨어 감염으로 인한 피해 최소화를 위한 긴급조치
        - 감염된 시스템을 외부 환경으로부터 격리화: 와이파이와 이더넷 케이블과 같은 네트워크 연결 해제, 외장하드와 같은 외부 저장장치 연결 해제
        - PC 전원은 끄지 않고 켜둔채로 둔다.
        - 암호화된 파일 백업 (복구의 잠재성과 포렌식 조사를 위함)

    3. 보안 당국에 신고
        - 감염 알림창이나 암호와 된 파일이 생성된 화면 캡쳐 이후 보안당국 제출
            e.g. 한국인터넷진흥원(boho.or.kr), 경찰청(ecrm.police.go.kr)
        - 해커와 협상을 먼저하거나, 해커에게 비용 지불은 금지.

    4. 데이터 복구
        - 랜섬웨어에 의해 암호화되지 않은 PC나 이동식 저장장치에 데이터를 백업.
        - 감염된 PC를 포맷 이후에 운영체제 재설치 및 재복구
        - 소프트웨어에 정기적인 최신 보안 업데이트 적용, 안티 바이러스/악성코드 프로그램 설치, 방화벽 설정 및 정기적인 백업 설정.
        - 암호화된 파일경우 랜섬웨어 복구 도구를 활용하거나 민간 보안업체 이용.
            e.g. 인터넷진흥원의 암호이용활성화, 노모어랜섬(No More Ransom)


#### 8. ARP 스푸핑에 대해 설명해주세요.
ARP는 Addres Resolution Protocol의 약자로서 IP주소를 MAC 주소로 변환해주는 프로토콜이면서 같은 네트워크상에 있는 장치끼리 의사소통을 할때 필요합니다.

그렇다면 ARP Spoofing은 근거리 통신망(LAN) 환경 안에서 주소 결정 프로토콜(ARP) 메세지를 이용해서 상대방의 데이터 패킷을 중간에서가로채는 중간자 공격 기법입니다.

공격자는 위조된 ARP 메세지를 근거리 통신망 환경에서 보내게 되는데, 이 위조된 메세지는 공격자의 MAC 주소를 네트워크에 있는 다른 합법적인 장치의 IP 주소와 연결하여 네트워크 내 장치들을 속여 송신지의 데이터 패킷을 원래 목적지 대신 공격자에게 보내도록 만듭니다. 

Case: 클라이언트 - 서버
공격자가 클라이언트가 서버에게 보내는 패킷을 먼저 받은 뒤, 서버에게 패킷을 보낸다.
- 즉, A와 B가 통신을 할 때 자신의 MAC 주소를 B의 MAC 주소라 속이고 B가 받을 패킷을 자신이 먼저 받고, B에게 그 패킷을 보내주는 방식.
- 특별한 이상 증상이 없기 때문에 공격당하는 것이 확인하기가 힘듬.

Man-in-the-Middle Attack (MITM) Attack 이라고 불리며
ARP 스푸핑의 목적은 민감한 정보의 도청, 로그인 세부정보와 같은 도영, 네트워크 트래픽에 악성코드 삽입 혹은 비인가적인 수행

예방 및 탐지 방법은 
    ARP 모니터링, 
    정적인 ARP 항목 사용, 
    ARP Spooficing 탐지 도구 배포,
    보안 네트워크 프로토콜 사용: e.g. ARP 스푸핑 탐지 소프트웨어, 암호화 네트워크 프로토콜
    정기적인 네트워크 모니터링.


#### 9. 피싱, 스미싱, 파밍 세 개의 특징과 차이점은 무엇인가요?
피싱(Phishing)
피싱은 사기성 이메일, 메세지 혹은 웹사이트를 통해 신뢰할만한 기관으로 가장하여 개인의 인증번호나, 신용카드번호, 계좌정보 등을 빼내 이를 불법적으로 이용하는 사기 수법입니다.

스미싱(Smishing)
스미싱은 SMS와 같은 문자 메시지 혹은 메시지 플랫폼을 통해서 발생하는 일종의 피싱입니다. 피싱과 동일하게 사용자들을 속여 개인정보를 유출하거나 악성 링크를 클릭하도록 유도합니다.
e.g. 무료쿠폰, 스마트 명세서와 같은 낚시성 광고 문자

파밍(Pharming)
파밍은 DNS 서버를 조작하여 피해자 컴퓨터의 호스트 파일을 손상시키거나 변조함으로써 사용자가 가짜 웹 사이트를 합법적인 진짜 웹사이트로 오인해 접속하도록 유도한뒤 개인정보를 탈취하는 기법입니다.
사용자가 정확한 웹사이트 주소를 입력했더라도, DNS 설정 혹은 호스트 파일이 변조가 되어있는 상황이기 때문에, 사용자가 인지하지 못하고 가짜 사이트로 접속이 되어짐.

피싱, 스미싱, 파밍 사이간 약간의 차이점이 있습니다.

[전달 방식]
피싱은 주로 이메일로, 스미싱은 SMS와 같은 문자 메시지, 파밍은 DNS로 통해 웹 트래픽을 우회시키는 방식입니다.

[실행 방식]
피싱과 스미싱은 사회 공학 방식을 의존 공격이라면,
파밍은 DNS 인프라나 로컬 컴퓨터 설정의 취약점을 이용하는 기술적인 공격입니다.


#### 10. APT 공격이 무엇인지 간략하게 설명해주세요.
APT는 Advnaced Persistent Threat의 약자로 지능형 지속 공격이라고 할수 있으며, 일반적으로는 APT는 타겟의 보안을 저해하거나 중요 정보를 유출하는 것을 목적으로 하며, 일반적인 사이버 공격과 달리, 특정 타겟을 정해 조직적이고 전문적으로 계획된 공격을 의합니다.

즉, APT는 하나의 공격 방식이라기보다는 원하는 목적 혹은 정보를 얻기위해서 악성코드, 피싱, 소셜 엔지니어링 등 다양한 공격방식을 이용해서 지속적으로 공격하는 방식이라고 생각하면 된다.

다양한 보안 위협으로 기업, 금융기관, 정부 등의 컴퓨터 시스템을 지속적으로 공격하는 것으로 주로 통신망을 통해서 시스템 내부로 침투해 공격.

APT는 사회관계망 서비스를 사용해서 정보수집, 악성코드 배포를 수행하고 공격 표적을 선정하여 지속적으로 공격을 수행.

APT의 특징으로는
- 일반적으론, 타겟 조직의 보안 시스템을 우회하거나 회피하는 기술 사용
- 일회성이 아니라 시스템에 잠복하여 지속적으로 공격
- 타겟 조직의 특적 APT는 특정한 타겟 조직을 선택하여 공격하는데, 타겟 조직은 국가, 기업, 정부 기관 등이 될수가 있다.
- 대부분 개인 혹은 기업에서, APT 공격을 받았을 경우 언제 어떤 경로로 악성코드가 침입하고 활동했는지 파악하는 것은 매우 어렵다. 사전에 미리 계획한 후 취약한 곳에 침두하여 공격을 하기 때문

APT 공격 단계
침투: Email, USB, 웹사이트를 통한 악성코드 등을 이용해 침투
탐색: Network 정보, 시스템 정보, 계정 정보 및 DB/시스템 정보 or 구조에 대한 정보 탐색
수집/공격: 목표로 한 데이터 수집, 시스템 공격
유출: 분석 및 추가 공격 혹은 금전적 이익을 취하기 위해 정보를 유츨합니다.


#### 11. SQL 인젝션 공격이란 무엇이며, 이를 방지하기 위한 스크립트 언어의 역할은 무엇인가요?
SQL Injection은 공격자가 악성 SQL 코드를 웹 애플리케이션의 입력 필드에 삽입할때 발생하는 공격기법입니다. 해당 공격은 애플리케이션의 데이터베이스 층의 취약점을 악용하여, 공격자가 인가받지 않은 SQL 명령을 실행할 수 있도록 만듭니다.

스크립트 언어의 역할은
Input Sanitization
    PHP, Python 과 같은 스크립트 언어는 입력 유효성 검사와 무효화를 통해 SQL Injection 공격을 방지하는데 중요한 역할을 함.

Prepared Statements/Parameterized Queryies
    개발자가 SQL 코드를 사용자의 입력에서 분리할수 있도록 Prepared Statement 혹은 Parameterized 쿼리 기능 제공.

    동적인 데이터와 바인딩 파라미터에 자리 표시자를 사용함으로써, SQL 쿼리 구조가 그대로 유지되어 공격자가 악성코드를 삽입 하는것을 방지할수가 있음.

ORMs(Object-Relational Mapping)
    스크립트 언어에서 제공하는 ORM이나 프레임워크는 SQL 데이터베이스의 상호작용을 투명하게 처리함.
    개발자가 직접적으로 SQL 쿼리를 쓰는 대신 데이터 접근 및 조작을 좀더 안전한 방식을 제공함으로써 SQL Injection 공격의 위험을 줄임.

Escaping Special Characters
    스크립트 언어는 사용자 입력 안에 있는 특수문자를 이스케이핑 할 수 있는 (보안 문자 처리를 할 수 있는) 함수 혹은 라이브러리를 제공합니다.
    잠재적으로 위험한 문자를 위험이 업슨ㄴ 표현식으로 전환해서 SQL Injection 공격의 위험을 줄임.


#### 12. 3 way-handshake와 4 way-handshake를 설명해주세요.
3 way-handshake는 TCP 연결 설정을 하는데에 사용되며 3가지의 단계가 일어납니다. SYN, SYN-ACK, ACK

1 단계, SYN: 클라이언트가 연결 요청을 하기 위해 SYN(동기화) 패킷을 서버에 보냅니다.
해당 SYN(동기화) 패킷에는 통신을 시작하기 위한 임의로 생성된 시퀀스 번호가 포함되어있습니다.

2 단계, SYN-ACK: 서버는 SYN 패킷을 클라이언트로부터 받은 이후로, 클라이언트에게 SYN-ACK(동기화 승인) 패킷을 전송하여 클라이언트로부터의 요청을 승인하게 됩니다.
해당 SYN-ACK 패킷에도 무작위로 생성된 시퀀스 번호가 포함되어있으며 클라이언트의 시퀀스 번호를 확인합니다.

3 단계, ACK: 클라이언트는 서버에서 보낸 SYN-ACK(동기화 승인) 패킷 수신을 확인하고, ACK(승인) 패킷을 전송합니다.
이 시점에서 TCP 연결이 설정되고 클라이언트와 서버 모두 데이터 전송을 시작할 수 있습니다.

4 way-handshake는 연결 설정된 TCP 연결을 동기화하고 순서대로 종료하는데에 사용되어 클라이언트와 서버간의 효울적인 통신과 종료를 보장하며, 4가지의 단계가 일어납니다. FIN, ACK-FIN, FIN, ACK

1 단계, FIN: 클라이언트 혹은 서버는 연결을 닫기 위해 FIN(완료) 패킷을 전송하여 종료를 시작합니다.

2 단계, ACK: FIN 패킷을 받은 수신자측은 (서버 혹은 클라이언트) FIN 패킷을 확인하고 연결을 닫을 준비가 되었다는 ACK(승인) 패킷을 보냅니다.

3 단계, FIN: 수신자측은 연결 종료를 알리기 위해 자체 FIN 패킷을 발신자측에게(처음 FIN 패킷을 수신자에게 보낸측) FIN 패킷을 보내 종료를 시작합니다.

4 단계, ACK: 미지막 단계로서, 4 way-handshake을 시작한 당사자는 상대방 측으로부터 FIN 수신을 확인하여 종료 프로세스를 완료하며 이 시점으로부터 연결은 완전히 닫힙니다.

#### 13. HTTP Method와 각각이 사용되는 경우에 대해서 설명해주세요.
GET
GET 메소드는 지정된 리소스에서 데이터를 요청하는데 사용됩니다.
서버의 상태나 데이터를 변경하지 않고 정보를 검색합니다.
웹 페이지, 이미지, 파일 혹은 기타 리소스를 가져오는데 사용합니다.
e.g. 유저가 브라우저에 URL을 입력하여 웹페이지에 접근할 때, 해당 페이지의 HTML 콘텐츠를 검색하기 위해 GET 요청 사용.

POST
POST 메소드는 데이터를 생성하거나 업데이트하기 위해 지정된 리소스에 데이터를 제출하는데 사용됩니다.
POST 메소드는 request의 body안에 데이터를 양식 제출, 파일 업로드 또는 사용자 입력을 하기 위해 서버에 보냅니다.
e.g. 온라인 양식을 작성하고 제출할 때 일반적으로 양식 데이터는 POST 요청을 사용하여 서버로 전송

PUT
PUT 메소드는 특정 URL에서 리소스를 업데이트를 하거나 교체하는 데 사용됩니다.
대상 리소스를 제공된 데이터로 바꾸기 위해서 request의 body안에 데이터를 보냅니다.
PUT의 여러개의 동일한 요청이 PUT의 단일 요청과 동일한 효과를 가지는 특징이 있으며 이를 멱등적인 특징을 가지고 있음을 알수있습니다.
e.g. 업데이트된 데이터를 PUT 요청을 사용하여 사용자의 프로필 URL로 전송해ㅅ, 서버내의 기존 사용자 프로필을 업데이트합니다.

DELETE
DELETE 메소드는 URL로 식별된 지정된 리소스를 삭제/제거 하는데에 사용됩니다.
해당 리소스를 영구적으로 제거할수 있도록 서버에 지시합니다.
e.g.
해당 리소스의 URL에 DELETE 요청을 보내서, 특정 항목을 제거하거나 사용자계정을 삭제합니다.

PATCH
PATCH 메소드는 리소스에 부분 수정을 적용하는데에 사용됩니다.
PUT 처럼 리소스를 완전히 교체하는 대신에, 리소스를 업데이트하기 위해 부분 수정 사항이 포함된 데이터를 보냅니다.
e.g
사용자의 프로필 사진을 업데이트하거나 특정 설정을 바꿀떄,
기존 리소스의 특정 속성이나 필드를 전체 리소스를 바꾸지 않고 수정하는 것

HEAD
HEAD 메소드는 해당 리소스의 헤더를 요청하는데에 사용됩니다.
리소스 자체를 검색하지 않고 헤더 등 요청된 리소스에 대한 메타데이터 정보를 검색합니다.
전체 콘텐츠를 가져오지 않고 업데이트를 확인하거나 리소스의 존재 여부를 확인하는 데 유요하게 쓰입니다.

OPTIONS
OPTION 메소드는 대상 리소스에 대한 통신 옵션을 설명하는데에 사용됩니다.
특정 URL에 대해 서버에서 지원하는 HTTP 메소드를 검색할수 있습니다.
특정 리소스에 대해서 어던 방법이 허용되거나 지원되는지 결정하는 데 도움이 됩니다.

TRACE
TRACE 메소드는 주로 진단 목적이나 디버깅의 목적으로 수신된 요청을 클라이언트에 다시 에코하는데에 사용이 됩니다.
클라이언트가 서버로 TRACE 요청을 보내게되면 서버는 수신된 요청을 다시 클라이언트로 에코하게 되는데, 중간에 프록시 또는 서버에 의해 변경된 내용을 확인할수 있도록 합니다.
요청 문제와 잠재적인 문제 진단이 가능하며, 공격자가 웹 취약성을 악용하여 중요한 정보를 수집하는 XST에 잠재적으로 악용이 될수가 있어서 최신 웹 서버는 종종 TRACE 메소드를 제한하거나 비활성화합니다.

CONNECT
CONNECT 메소드는 주로 프록시 서버를 통해 지정된 서버에 대한 연결을 설정하는 데 사용됩니다.
클라이언트가 서버와의 안전한 암호화 연결을 HTTP 프록시를 통해 연결하는 상황에서 사용이 됩니다.
e.g. 프록시 서버의 HTTPS를 통해 웹사이트를 접근하는 경우, 클라이언트와 서버간의 보안 통신을 활성화할때 필수적인 메소드


#### 14. www.naver.com에 접속할 때 생기는 과정에 대해 설명해주세요. (웹 동작 방식 이해)
유저가 웹 브라우저 주소 바에 'www.naver.com'을 입력하고 엔터를 치게 된다면, 브라우저는 도메인 이름인 'www.naver.com'을 IP 주소로 확인하기 위해 브라우저 쪽의 캐시를 확인하게 됩니다. 캐시쪽에서 해당 도메인에 대한 IP 주소 정보가 없다면, 브라우저는 DNS query를 DNS 서버쪽으로 보내게 됩니다.

DNS 서버는 도메인 이름을("www.naver.com") IP 주소로 변환하여, 브라우저가 해당 웹사이트를 호스팅하는 서버를 찾을수 있게 해당 IP 주소를 브라우저에게 반환하게 됩니다.

브라우저가 IP 주소를 획득한 이후, DNS 서버 혹은 캐시에서 검색된 주소 IP와 연결된 서버쪽으로 HTTP 요청(PORT 80) 혹은 HTTPS 요청(PORT 443)을 시작하게 됩니다. 이 요청의 경우 서버가 요청을 처리하는 데에 필요한 HTTP 메소드 헤더(e.g. GET, POST ...) 및 추가적인 데이터가 포함되어 있습니다.

'www.naver.com'을 호스팅하고 있는 서버는 해당의 HTTP 요청을 받고 그 요청을 처리하게 됩니다.
서버에서는 요청된 웹페이지나 컨텐츠를 서버의 저장소에서 조회해서, 적절한 응답을 생성한뒤에, 그 결과 페이지와 컨텐츠가 포함된 응답을 유저의 브라우저로 보내기위한 준비를 합니다.

그 이후에 서버는 응답 상태, 컨텐츠 타입, 캐싱 지시문 등에 대한 정보를 제공하는 헤더와 함게 요청된 웹 페이지 데이터를 포함하고 있는 HTTP 응답을 브라우저의 요청에 따라 브라우저에게 응답을 하게 됩니다.

응답을 받은 이후에, 유저의 브라우저는 웹페이지를 랜더링을 시작합니다. HTML 구문을 분석하고, Javascript 혹은 CSS 파일을 실행하여, HTML 내에 참조되는 이미지, 스타일시트 혹은 스크립등과 같은 추가 리소스들을 가져옵니다.

브라우저는 가져온 리소스들을 조합하고 HTML, CSS 및 Javascript의 설정에 따라 웹페이지를 랜더링하게 됩니다.

랜더링을 하는 동안, 브라우저는 HTML 내에 연결된 추가적인 리소스들(이미지, 비디오, 또는 외부 스크립트들)을 접할수가 있는데, 각 리소스에 대해서는 위에서 언급한것과 같이(서버에게 HTTP/HTTPS 요청 -> 서버 처리 및 서버의 HTTP/HTTPS 응답) 해당 특정 리소스에 대해 각 서버와 새 요청을 시작합니다.

그 이후 사용자는 브라우저 창에 "www.naver.com"의 완전히 랜더링된 페이지를 볼수있게 되면서 해당 웹사이트와 상호작용(링크 클릭, 폼 제출 등등...) 이 가능하게 되는데, 그 이후의 HTTP 요청들은 또 다시 서버에 보내지고, 서버에서 처리 이후 새로운 요청에대해 응답하는 사이클을 계속 돌게 됩니다.


#### 15. VPN이란 무엇입니까?
VPN은 Virtual Private Network의 약자로, 공용 인터넷 연결 환경을 통해 안전하고 암호화된 사설 네트워크를 생성하는 기술입니다. 이를 통해서 사용자는 사용자의 장치와 원격 서버 혹은 네트워크 사이로 안전하게 접근이 가능하고 안전하게 데이터를 전송할수 있습니다.

[암호화와 터널링]
VPN은 사용자 장치와 VPN 서버간에 보안 터널을 생성하게 되는데, 이 터널을 통해 전송 되는 모든 데이터는 암호화 프로토콜로도 암호화 되기떄문에 해커나 승인되지 않은 유저가 민감한 정보에 무단으로 접근하거나 가로채는 것을 방지할수 있습니다.

[익명성과 개인 정보 보호]
또한 VPN은 익명성을 보장하고 개인 정보를 보호할수가 있습니다. VPN 서버를 통해서 인터넷 트래픽을 라우팅하기떄문에, VPN은 사용자의 IP 주소를 마스킹하여 사용자의 연결이 VPN 서버 위치에서 시작된것처럼 보이게 합니다. 이를 통해 웹사이트 혹은 서비스가 사용자의 실제 위치나 신원을 추적하는것을 방지합니다.

[접근 제어 및 제한 우회]
VPN을 통해서 사용자는 지리적 제한을 우회할수가 있기 때문에, 해당 지역에서 차단 혹은 제한되는 콘텐츠나 서비스를 접근이 가능합니다.
마치 사용자가 VPN 지역에 있는것처럼 웹사이트/서비스 접근 할수가 있습니다.

[원격 액세스 및 보안 연결]
VPN을 사용하면 회사 직원 경우 회사 네트워크에 원격으로 안전하게 접근이 가능합니다. 즉, 기밀성과 보안을 보장하면서 안전하게 회사 리소스와 시스템에 접근할수 있도록 합니다.

[공용 WI-FI 보안]
보안 수준이 낮은 카페와 공항같은 공용 와이파이 네트워크 환경과 같은 곳에서, VPN은 데이터를 암호화하고 데이터가 유출되거나 무단 접근의 위험을 줄여 보안성을 높여줍니다.
